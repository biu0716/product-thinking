# 为什么我把 PR Agent 拆成四个模式，而不是做一个聊天框

我做 PR Agent 的时候，一开始做的就是一个聊天框。

逻辑很简单：我每天用 ChatGPT 和 Claude 干活，觉得如果能加上行业知识库，效果应该更好。所以最初的想法就是做一个"带知识库的聊天机器人"，用户上传资料，然后在对话框里提需求，AI 结合资料生成内容。

做出来之后我发现一个问题：如果还是一个聊天框，我跟 ChatGPT 有什么区别？

我自己用 ChatGPT 干活的时候就感觉很麻烦。一篇稿子从生成到定稿，我可能一会儿让它整合信息，一会儿让它改错别字，一会儿让它换个风格。这些任务混在同一个对话里，几轮之后就开始乱了。

最典型的问题是：我先上传了一份参考稿让 AI 模仿风格，中间又加了几轮新的需求，聊了几轮之后 AI 就逐渐偏离了最初的风格。关键是它不会告诉你"我忘了要模仿什么风格"，它只会悄悄偏离，还得你自己发现。而且想回头找上一版的输出，在一长串对话里翻起来也很费劲。

效率很低，体验跟直接用 ChatGPT 没有本质差别。

我开始想，我的日常工作到底可以拆成哪几类任务？

---

想清楚之后，答案是四个：generate、check、normalize、rewrite。

**Generate**，对应的是从零生成。我手上有一堆散装资料——车型参数、活动信息、领导发言，需要把它们整合成一篇新闻稿。这是最常见的任务，也是大多数人对"AI 写作"的第一印象。

**Check**，对应的是检查。稿子写完之后，需要看有没有事实性错误、有没有错别字、参数有没有写反。这件事听起来简单，但在汽车 PR 里出一个参数错误是很严重的事。

**Normalize**，对应的是规范化。如果说 Check 像是语文老师改错别字，Normalize 更像是帮你把方言换成普通话——内容没有"错"，但表达不够规范。这个模式比我预想的更有用。有些表达在行业内部说习惯了，完全不觉得有问题，但对外发布时不够规范。比如我们会说"德国时间"，但正式稿件里应该写"中欧时间"。我在行业里泡了四年，这种说法我听了一百遍，已经完全不觉得它有问题了。Normalize 的价值在于充当一个行业外的眼睛——帮你跳出行业视角审视自己的表达。

**Rewrite**，对应的是风格改写。内容本身没有错误，但需要调整表达方式——"写得更 social 一点""更直接一点""不要爹味"。这恰好是我在另一篇文章里讨论的问题：这类需求往往是模糊的，AI 最容易理解偏。

---

拆成四个模式之后，产品层面最大的变化是：**用户不需要自己判断"我现在该怎么跟 AI 说话"了。**

在通用聊天框里，用户得先想清楚自己要什么，然后把需求翻译成 AI 能理解的 prompt。这个翻译过程本身就是负担。而在 PR Agent 里，用户进来选一个模式，产品就已经帮他做了第一层判断——你现在需要的是生成、检查、规范化还是改写。后续的 prompt、规则约束、输出格式都是模式预设好的，用户只需要提供素材。

这四个模式背后的输入形态、规则依赖和输出期望完全不一样。Generate 需要大量素材输入，输出是一篇完整的稿件。Check 的输入是一篇已经写好的稿件，输出是一份错误清单。Normalize 的输入也是成稿，但它找的是"不够规范"的表达，跟 Check 找错误的判断标准完全不同。Rewrite 的输入是成稿加上一个风格指令，输出是同样内容的不同版本。

把这四种任务塞进一个聊天框，AI 需要先理解用户想做哪种，再切换到对应的执行逻辑。这等于把产品经理该做的判断甩给了用户和模型。拆开之后，每个模式有自己的规则、自己的 prompt 模板、自己的输出格式，可控性提高了很多。

---

这个拆分过程给我最大的收获，其实是让我对自己的工作理解变得更清晰了。

我做了四年 PR，但在做 PR Agent 之前，我其实从来没有仔细想过"我的工作到底可以怎么划分"。每天接到任务就做，写稿就写稿，查错就查错，改风格就改风格，它们混在一起，感觉就是"写东西"。直到我需要把这些任务变成产品功能的时候，我才被迫去想：这些任务的本质区别是什么？它们的输入和输出有什么不同？为什么不能用同一套逻辑处理？

产品化的过程反过来帮我理解了自己的业务。

我觉得这可能是所有垂直场景 AI 产品的起点：先把你自己的工作拆清楚。你每天做的事情里，哪些是重复的、哪些是有规则的、哪些是模糊的、哪些是只有你能判断的——想清楚这些，产品的形态就自然出来了。
